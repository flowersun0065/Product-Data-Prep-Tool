> 修复时间：2026-05-10

## 零、面板展示与数据源不一致类

### 问题 0：确认入库后，面板又被旧数据覆盖显示

**场景**：用户在品牌面板里逐一修正了多个商品的品牌（例如把 5 个待缺失商品的品牌从"可比克"
改为"乐事"），侧边栏和状态标签都显示"已设置"。然后点击"确认入库"完成导出，弹出提示成功后
关闭弹窗，回到品牌面板一看——刚才那些商品又显示回"建议品牌: 可比克"，好像没改过一样。

**原因**：系统内部维护了两套数据：

- `diagnosisData.brand_clusters[].items[]` 中的 `item.brand` / `item.suggested_brand`——诊断时的**原始**值，**从不更新**
- `brandRules[code]` 中的 `rule.brand`——用户修正后的值，**实时更新**

所有渲染函数（分组卡片 `renderGroupCard`、详情面板 `renderPanelItem`）显示品牌名称时读的都是第一套
原始数据。用户修正后的品牌只出现在一个小的绿色状态标签里。导出品牌库后 `confirmExportToLibrary()`
仅刷新了 `brandRules` 数据，没有触发任何 UI 重新渲染。当用户下次切换视图或打开侧面板时，
渲染函数重新执行，从原始 `diagnosisData` 里取数据，自然显示回修改前的旧值。

**修复**：

1. `renderPanelItem()` 增加判断：如果 `brandRules[code]` 中存在用户修正的品牌，主品牌行优先
   显示修正后的值（绿色"已设置品牌: xxx"），不再显示原始 AI 建议
2. `renderGroupCard()` 卡片头部同样检查：该组内所有商品是否已被修正。如果全部修正为同一品牌，
   卡片头部直接显示"已设置品牌: xxx"；部分修正的显示进度；约定品牌修正为多个不同品牌的注明数量
3. `confirmExportToLibrary()` 在同步数据和清空前端的确认品牌后，主动触发 `renderBrandGroups()` 和
   `renderPanelContent()`，确保面板在导出后立刻刷新为最新状态

**结果**：面板上所有品牌名称会实时反映用户的修正操作，导出后不会出现"已修正又变回去"的
现象。

### 问题 1：处理后刷新页面，诊断结果全部消失

**场景**：上传了一份 500 行的小文件，系统瞬间出结果。处理完几个品牌后刷新了一下浏览器
页面，所有诊断结果都没了，必须重新上传文件。

**原因**：小文件走的是同步处理路径，诊断结果只存在于 HTTP 响应里，没有存到后端 session
中。刷新后前端丢掉了内存数据，后端也没存，无法恢复。

**修复**：同步路径补全了 session 持久化，诊断结果同时写入后端。新增了浏览器
localStorage 记录上次 session ID，刷新页面后自动尝试恢复。

**结果**：刷新页面后自动恢复当前的工作进度，无需重新上传。

### 问题 2：编辑了大量品牌信息，重启服务后全丢了

**场景**：诊断完成后花了 20 分钟逐一修改了 30 个新品牌的名称、类型、国家。还没来得及
点击"入库"，Flask 进程被重启了。重新打开页面导入同一个文件，所有编辑全部消失。

**原因**：`brand_rules` 虽然每条都写入了 `corrected_products.json`（可跨 session 恢复），
但侧边栏里 `new_brands` 的编辑（改名、改类型、改国家、未确认状态）没有任何持久化
路径。snapshot 只在诊断完成时保存一次。

**修复**：增加定时后台刷写机制（30 秒间隔），每次用户编辑品牌规则、添加品牌、忽略品牌、
保存分类规则时都触发标记。snapshot 写入也改为了原子写入（先写临时文件再替换），防止
进程崩溃时文件损坏。

**结果**：用户操作即时被保护，即使忘记入库或进程崩溃，重启后也能恢复所有编辑。

---

### 问题 3：添加了品牌但实际没保存到后端

**场景**：在侧栏添加了一个新品牌"好想你"，前端列表中确实出现了。然后给几个商品设置了
该品牌。刷新页面后发现这些商品仍显示"品牌缺失"，"好想你"也不在侧栏中。

**原因**：前端 `addNewBrand` 函数先把品牌推到本地数组，再发请求给后端。如果后端请求
失败（网络超时、后端错误），前端本地已经有了这个品牌，用户察觉不到异常，后续操作都
基于一个"虚拟存在"的品牌。

**修复**：改为先调后端，请求成功后才更新前端状态。失败时弹出明确的错误提示，不更新
本地数据。

**结果**：添加的品牌一定是后端确认成功的，不存在"前端有、后端无"的幽灵品牌。

---

### 问题 4：侧边栏修改品牌信息后，下次看到的是旧值

**场景**：在侧栏将品牌类型从"未知"改为"零食"，前端显示已更新。但之后再打开同文件，
类型又变回了"未知"。

**原因**：同问题 3 的模式——`submitNewBrandFromModal` 先更新前端 `newBrands[index]`，
再发后端请求。请求失败时只打了一条 `console.warn`（用户完全看不到），前端已显示新
值而 session 里是旧值。

**修复**：先调后端确认成功，再更新前端。失败时弹出提示"品牌信息保存失败，请重试"，
不关闭弹窗。

**结果**：修改的品牌信息保证前后端一致。

## 二、数据不正确/过期类

### 问题 5：预览页面显示的与品牌面板操作的不一致

**场景**：在品牌分组面板里把 5 个商品的品牌手动标记正确后，切换到"全量预览"页，发现
那 5 个商品仍然显示"品牌错误"。

**原因**：预览接口 `get_preview()` 重新读取原始 Excel 文件，对每行用文件里的原始品牌
值做一致性检查，完全没读用户已保存的 `brand_rules`。

**修复**：预览遍历时，先检查该商品 code 是否有已保存的品牌规则，优先用规则覆盖原始
品牌值再进行检查。

**结果**：预览页展示的是用户操作后的最新状态，与品牌面板保持一致。

---

### 问题 6：诊断进度跑完了但结果还是空的

**场景**：上传了一个大文件，进度条走到 100% 显示"诊断完成"，前端自动请求诊断结果，
但返回的数据为空或不完整。

**原因**：诊断后台线程先设置了 `diagnosis_result`，隔了几行代码后才改
`diagnosis_status = 'completed'`。前端每 3 秒轮询状态，如果轮询恰好落在两次赋值之间，
会读到 `status=completed` 但 `diagnosis_result` 还没写入（或读到了
`diagnosis_progress` 相关的中间状态）。

**修复**：所有结果数据先计算好，然后在一个锁保护的原子操作里一次性写入：先写
`diagnosis_result`、`diagnosis_stats`、`diagnosis_progress`，最后改 `status`。前端
见到 `completed` 时数据一定就绪。

**结果**：诊断完成后前端拿到的结果是完整、一致的。

## 三、文件损坏/截断类

### 问题 7：处理大文件时下载的 Excel 打不开

**场景**：处理一个 5000 行的文件，看到处理进度在走，随手点了"下载结果"，拿到的
xlsx 文件打开报错"文件已损坏"或行数明显不对。

**原因**：`process_file_async` 每处理完一批（20 条）就覆盖写入同一个结果 Excel。
`pd.to_excel()` 内部是先打开文件、写内容、关闭文件——如果用户在上一步和下一步之间
（文件刚被打开、内容还没写完）点了下载，就会拿到一个残废文件。

**修复**：所有 Excel 写入改为先写到临时文件（`.tmp`），写完后用 `os.replace()` 原子替换。
操作系统保证 `rename` 是原子的——要么拿到旧完整文件，要么拿到新完整文件，不存在中间态。

**结果**：任何时候下载都能拿到完整可打开的 Excel 文件。

---

### 问题 8：进程崩溃后缓存文件全部损坏

**场景**：系统在写入 `corrected_products.json` 时进程被 kill -9 或突然断电。重启后该
JSON 文件内容为空或格式错误，导致之前积累的所有修正记录全部丢失。

**原因**：所有 JSON 持久化（cache、corrected_products、session_snapshots、
dismissed_brands、brand_config 等）都是直接 `json.dump` 覆盖原文件。如果写入过程中
进程崩溃，原文件被截断为乱码或空文件。

**修复**：统一改为先写临时文件（`.tmp`）、写完整后 `os.replace()` 替换原文件。即使写入
中途崩溃，也只是留下一个 `.tmp` 垃圾文件，原文件完好无损。

**结果**：所有持久化数据有崩溃保护，最坏情况只丢最后一次未完成的操作。

## 四、服务稳定性类

### 问题 9：同时做两个操作导致数据互相覆盖

**场景**：在品牌面板快速点选多个商品设置品牌规则时，有时出现了两次操作结果互相覆盖
的情况，后一次操作的旧值覆盖了前一次操作的新值。

**原因**：`sessions` 字典被 Flask 的 HTTP 请求线程和诊断后台线程同时读写，没有任何
锁保护。Python 字典的单个赋值操作在 CPython GIL 下是安全的，但"先读再写"的组合操作
（如 `session['logs'].append(...)`、`session['brand_rules'][code] = ...` ）不是
原子的。

**修复**：新增 session 级别的锁机制 `_get_session_lock()`，所有涉及读-改-写的操作都用
锁包裹。包括品牌规则保存、诊断日志追加、新品牌列表修改等。

**结果**：多线程并发操作不会互相覆盖。

---

### 问题 10：同一秒上传两个文件，后一个覆盖前一个

**场景**：快速连续导入两个历史文件，发现第一个文件的 session 数据完全被第二个覆盖，
无法再访问第一个的诊断结果。

**原因**：session ID 用 `int(time.time())` 生成，精度只到秒。同秒内两次上传生成相同
ID，后创建的直接覆盖。

**修复**：session ID 改为 `session_{timestamp}_{uuid前6位}` 格式，加入随机后缀确保
唯一性。

**结果**：即使毫秒级连续上传也不会碰撞。

---

### 问题 11：诊断过程中重复导入同一文件

**场景**：在"最近文件"列表里点了一次导入，等待诊断时以为没反应又点了一次。结果同一个
文件被诊断了两次，两个后台线程都在更新状态，日志混乱。

**原因**：`import_recent_file` 不检查该文件是否已有正在诊断的 session。

**修复**：创建新 session 前先遍历已有 session，如果发现相同文件路径且 status 为
`processing`，直接返回已有 session ID，复用当前诊断。

**结果**：不会重复触发诊断，节省资源。

---

### 问题 12：服务器宕机时前端无限弹错误

**场景**：Flask 进程意外退出，浏览器页面还在。进度条区域不断闪烁"获取状态失败"，每
10 秒弹一次错误，永不停止。用户不知道是服务器挂了还是网络问题，只能手动关页面。

**原因**：`pollDiagnosisStatus` 的 `catch` 分支无条件 `setTimeout` 重试，没有上限。

**修复**：增加重试计数器，最多重试 5 次。每次间隔指数退避（10s→20s→40s→60s→60s）。
达到上限后显示"服务器连接失败（已重试 5 次），请刷新页面重试"并停止。正常响应时
重置计数器。

**结果**：服务故障时有明确的停止和提示，不无限重试。

---

### 问题 13：品牌设置后前端状态可能被覆盖回旧值

**场景**：在品牌分组里快速对多个商品设置品牌，有时刚设置好的商品又显示回"品牌缺失"
状态。

**原因**：`saveBrandRule` 每次保存单条规则后都调 `syncBrandRules()` 做全量同步。当
用户快速点击多条时，多个 syncBrandRules 请求并发发出，响应顺序不保证。后发出的请求
如果先返回，会用旧数据覆盖刚收到的新数据。

**修复**：单条保存改为乐观更新本地 + 失败回滚模式，不再每条都调全量同步。全量同步
仅保留在关闭侧面板、批量保存完成、确认入库等合适的汇总节点。

**结果**：快速连续操作不会乱序，状态始终准确。

---

### 问题 14：批量操作时可能静默失败

**场景**：在分类面板点"全部确认"按钮，批量确认了几十个商品分类。操作完成后发现
部分商品没有被确认，页面没有任何错误提示。

**原因**：`batchConfirmAllCategoryItems` 中获取路径时写了
`item.all_paths[0]`，当 `all_paths` 为 `undefined` 时会抛 TypeError。这个错误被
循环外的 catch 吞掉（当时该函数没有 try-catch），整个批量操作中途崩溃但用户无感知。

**修复**：改用 `item.suggested_path?.[0] ?? item.all_paths?.[0] ?? ''`，为空的商品
安全跳过并打印 warn 日志。

**结果**：批量操作不会因个别异常数据而崩溃，全部能确认的都会被确认。

## 五、安全类

### 问题 15：API Key 暴露在浏览器地址栏和日志中

**场景**：填写 API Key 后点击"导出预览"，新打开的窗口地址栏里赫然出现了 Base64 编码
的 API Key，任何站在这台电脑前的人都能看到。

**原因**：`export.js` 用 `btoa(api_key)` 把 key 编码后放到 URL 参数里通过
`window.open()` 打开。Base64 不是加密，key 会出现在浏览器地址栏、历史记录、服务端
access log、HTTP Referer 头等地方。

**修复**：改为用 `sessionStorage` 临时存储 key（同源标签页间可共享，页面关闭自动清除），
不再进入 URL。

**结果**：API Key 不出现在任何可持久化或可被日志记录的地方。
